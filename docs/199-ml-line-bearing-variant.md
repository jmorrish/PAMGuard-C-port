# ML Line Bearing Variant

Date: 2026-07-21

## Purpose

`docs/194` recorded one branch of `BearingLocaliserSelector` the port did not reproduce: a line sub-array of more than two hydrophones uses `MLLineBearingLocaliser2` **when `SMRUEnable.isEnable()`**, and noted that "an SMRU-enabled PAMGuard would differ from this engine on a multi-hydrophone line." This closes that gap on the maths side.

## Reference semantics ported

`MLLineBearingLocaliser2` is a 51-line subclass of `MLGridBearingLocaliser2`, and essentially all of it is one method:

```java
protected double thetaBinToAngle(double bin) {
    return Math.PI/2. - super.thetaBinToAngle(bin);
}
```

The important part is **where that method is called**. `prepare` uses it while building the delay table, and Java dispatches virtually, so overriding it changes the table itself — not just the angle reported at the end. Treating this as a post-hoc output transform would give different numbers, and the fixture confirms it: `line-broadside` reports the same theta under both conventions but a **different** theta error, because the peak lands in a different bin of a differently-built surface.

So the port makes it a config flag consumed inside `theta_bin_to_angle`, which is the one place both the table build and the output read.

### A reference bug, recorded not reproduced

The subclass also overrides `thetaAngleToBin`:

```java
protected int thetaAngleToBin(double theta) {
    return thetaAngleToBin(Math.PI/2. - theta);
}
```

That calls **itself**, not `super`, so it is unbounded recursion and would overflow the stack on the first call. It never fires because its only caller is `localiseByCrawl`, which `localise` does not use — `fullGridSearch` does. The port has no equivalent method, so there is nothing to reproduce; it is noted here because a future port of the crawl path would inherit a defect, not a behaviour.

## Implementation

`MlGridBearingConfig::line_theta_convention`, default false. The header explains the SMRU gate so a reader does not assume the engine simply forgot to select it.

**Not wired into the selector.** `BearingLocaliserSelector`'s SMRU branch depends on a licence flag the engine has no notion of, and guessing would mean the open-source default build and this engine disagreeing. `select_bearing_localiser` continues to return `Pair` for a line sub-array, matching the default build exactly. The variant exists so an SMRU-licensed deployment can be reproduced when someone needs to compare against one — not as a path the engine reaches on its own.

## Validation

`ml_line_bearing_parity` (new) replays `tests/fixtures/localisation/ml-line-bearing.csv`, generated by the same exporter under a `line` mode argument so both conventions come from one transcription. Four line cases: broadside, oblique, endfire, and a four-hydrophone line with speed-of-sound error. Tolerance and comparison are shared with the grid check, which now takes the same mode argument.

The two fixtures discriminate: `line-endfire` and `line-four-oblique` do not appear in the grid fixture at all, and the cases that do appear in both report different theta errors under the two conventions, so a flag that was quietly ignored would fail this test rather than pass it.

Generating the line fixture left `ml-grid-bearing.csv` byte-for-byte unchanged, confirmed by `git status` — the exporter refactor did not disturb the pinned grid rows.

Full CTest suite passes `71/71`.

## Claim boundary

The maths is ported; the **selection** is not, and deliberately. Nothing in the engine chooses this variant, so an SMRU-licensed PAMGuard and this engine still differ by default on a multi-hydrophone line array. Closing that would mean adding a licence-equivalent switch to session config, which is a product decision rather than a porting one.

`thetaAngleToBin` and the crawl search path it serves remain unported, as `docs/195` already recorded for the base class.
