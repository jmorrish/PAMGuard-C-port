# Click Trigger Edge Fixture Sweep

Date: 2026-07-03

## Purpose

The click trigger had a single PAMGuard Java parity fixture (`basic-2ch-threshold10.csv`), so trigger parameter semantics beyond that one configuration were only covered by C++-side edge tests. The parity ledger's top priority was expanding Java fixture coverage across trigger parameter combinations. This slice adds six PAMGuard-Java-generated fixtures that pin down min-separation merge/split, max-length truncation, min-trigger-channel gating, and alternate threshold/filter constants.

## Exporter scenarios

`reference-tools/java/src/org/pamguard/port/reference/ClickTriggerFixtureExporter.java` now accepts an optional trailing `scenario` argument selecting the synthetic input signal:

- `single-transient` (default): transient at samples 80-86, channel 0 scale 1.0, channel 1 scale 0.82. Identical to the previous hardcoded signal; the 13-argument form regenerates `basic-2ch-threshold10.csv` byte-for-byte.
- `double-transient`: transients at samples 60-66 and 90-96 with the same per-channel scaling.
- `long-transient`: one transient spanning samples 60-140.
- `single-channel-transient`: transient at samples 80-86 on channel 0 only; channel 1 carries background noise only.

The exporter still drives the real PAMGuard `clickDetector.TriggerFilter` class for all filter maths.

## New fixtures

Generated by `reference-tools/scripts/generate-click-trigger-edge-fixtures.ps1` into `cpp-engine/tests/fixtures/click-trigger/`. All use channel/trigger bitmap `0x3`, pre 10, post 12, sample rate 48000 Hz; other parameters per row:

| Fixture | Scenario | Parameter under test | Encoded behaviour |
| --- | --- | --- | --- |
| `double-split-minsep8.csv` | double-transient | `minSep` 8 | Two separate clicks. |
| `double-merge-minsep16.csv` | double-transient | `minSep` 16 | One merged click spanning both transients. |
| `long-maxlen64.csv` | long-transient | `maxLength` 64, chunk 1024 | One click with duration clamped to exactly 64. |
| `single-channel-min1.csv` | single-channel-transient | `minTriggerChannels` 1 | One click with `triggerBitmap` `0x1`. |
| `single-channel-min2-suppressed.csv` | single-channel-transient | `minTriggerChannels` 2 | Zero detections (gate suppresses the single-channel trigger). |
| `single-threshold6.csv` | single-transient | threshold 6 dB, short filter 0.2, long filter 0.0001 | One click with different extraction window and signal excess than the basic fixture. |

The `long-maxlen64` fixture uses a 1024-sample block because trigger filters are initialised from the whole-block mean; an 81-sample transient inside a 256-sample block raises the background estimate enough that the 10 dB threshold is never crossed. This block-mean initialisation sensitivity is itself PAMGuard-faithful behaviour worth remembering when designing fixtures.

## Checker changes

`cpp-engine/tools/click_trigger_fixture_check.cpp` now accepts the full trigger configuration, sample rate, chunk length, scenario, and an optional expected detection count on the command line (defaults preserve the original basic-fixture invocation). It additionally asserts:

- The fixture encodes the intended scenario behaviour (expected detection count), so a silent regeneration regression cannot go unnoticed.
- Every fixture row satisfies `duration <= maxLength`, validating the truncation invariant generically.

The existing negative-config, trigger-gating, waveform-capture, and reset-reproducibility checks run unchanged for every fixture invocation.

## Validation

- `generate-click-trigger-fixture.ps1` (unchanged 13-argument path) regenerates `basic-2ch-threshold10.csv` with no diff.
- Seven `click_trigger_*` CTest cases pass, comparing sample indices, durations, channel/trigger bitmaps, and times exactly, and signal excess within `1e-10` dB.
- Full CTest suite passes `47/47` on Windows Release, including service smokes.

## Claim boundary

These fixtures cover single-block processing with block-mean filter initialisation, matching the exporter's reference loop. Cross-chunk trigger initialisation (first-chunk mean in the streaming C++ engine) and PAMGuard's dual-alpha long-filter path remain outside fixture coverage.
