# Earth-Frame Vectors

Date: 2026-07-22

## Purpose

`docs/197` and `docs/198` served direction vectors in the hydrophone array's xyz frame and recorded the earth frame as blocked on a GPS/attitude feed. That was too absolute. The **time-varying** case is blocked; the rotation itself is not, and neither is the **static** case — an array whose orientation is known and fixed for the session, which is exactly what a bottom-mounted or moored deployment has.

This ports `getRealWorldVectors` and serves earth-frame vectors when the session declares an array orientation.

## Reference semantics ported

`AbstractLocalisation.getRealWorldVectors` is short:

```java
GpsData origin = getOriginLatLong();
if (origin == null) {
    return v;
}
PamQuaternion direction = origin.getQuaternion();
for (int i = 0; i < v.length; i++) {
    v[i] = PamVector.rotateVector(v[i], direction);
    if (getSubArrayType() == ArrayManager.ARRAY_TYPE_LINE) {
        v[i].setCone(true);
    }
}
```

Two things make it a small port rather than a new one:

- **`GpsData.getQuaternion()` is `new PamQuaternion(toRadians(heading), toRadians(pitch), toRadians(roll))`** — byte-for-byte the same construction streamer orientation uses, so the quaternion maths ported and fixture-pinned in `docs/193` is the whole of the rotation, including the clockwise-heading and pitch-roll-heading conventions.
- **A null origin returns the vectors unrotated.** The engine's equivalent of "no GPS" is "no declared orientation" — with one difference, below.

The line-shape cone forcing after rotation is ported as written, though the line branch's vectors are cones already.

## The one deliberate difference

Where PAMGuard silently returns *unrotated* vectors when it has no origin, the engine reports **nothing**: no `array.orientation`, no earth-frame fields. Returning array-frame numbers under an earth-frame name is precisely the confusion `docs/196` refused to create by not calling theta an azimuth, and repeating the reference's fallback here would reintroduce it. A consumer who sees `earthWorldVectors` can rely on what the name says.

## Configuration and output

Session config gains `array.orientation` — `headingDegrees`, `pitchDegrees`, `rollDegrees`, all finite-checked. This is the **whole array's** orientation in the world (PAMGuard's vessel/GPS attitude), distinct from per-streamer orientation (`docs/193`), which positions streamers *within* the array frame and is already folded into hydrophone coordinates.

Schema v21 adds, when an orientation is declared:

- `earthWorldVectors` on `gridBearing` and on `lsqBearing`;
- `pairBearingEarthWorldVectors` on delay entries carrying a pair bearing.

Each mirrors its array-frame counterpart in length, order, and cone flags.

## Validation

`earth_vector_parity` (new) replays `earth-vector.csv`, generated by the world-vector exporter in an `earth` mode that drives the **real** `PamQuaternion` and `PamVector.rotateVector` over the same twelve cases. Every case gets a *distinct* orientation derived from its name via Java's `String.hashCode` — reproduced exactly in the C++ check, including the signed-overflow behaviour — so a dropped rotation cannot pass by coincidence and the two sides cannot drift apart through a copied table. All twelve match with `max_abs_error` **0**, as does the unchanged array-frame fixture.

`session_lsq_bearing_wiring` gains: no declared orientation produces no earth vectors; a declared 42°/-7°/11° orientation produces earth vectors matching the array-frame count, still unit length, and **measurably rotated** — a wiring that quietly dropped the orientation would fail that last check, not pass it.

Full CTest suite passes `72/72`.

## Claim boundary

**Static orientation only.** One heading/pitch/roll for the session's lifetime. A vessel under way, or a towed array changing attitude through a tow, needs the time-varying feed that remains genuinely blocked — this slice narrows that ledger item to its true scope rather than closing it.

Earth frame here means "the array frame rotated by the declared orientation". Geographic position (`LatLong`, and everything `addDistanceMeters` does with it) is not modelled; the engine still has no notion of where on the planet the array is, only which way it faces.

The whistle region-level `bearing` object and the far-field click bearings are unchanged; earth vectors attach only where array-frame vectors already existed.
