# Click Classifier Case Fixture Sweep

Date: 2026-07-03

## Purpose

The basic click classifier had one PAMGuard Java fixture pinning a single decision (two types, second matches). Decision semantics per criterion — and edge behaviours like the zero-max-length skip — were only covered by C++-side tests, which cannot prove PAMGuard parity. This slice expands the fixture to an eleven-case catalogue generated by the PAMGuard Java reference maths.

## Case catalogue

`ClickBasicClassifierFixtureExporter.java` and `click_basic_classifier_fixture_check.cpp` share a case catalogue by name; both sides build identical type lists per case, all derived from the same base type (species 42, all five criteria enabled). The exporter classifies the shared synthetic two-channel 96-sample waveform per case and writes `case,clickType,discard` rows; the C++ check replays every case and compares decisions exactly.

| Case | Deciding behaviour | Java result |
| --- | --- | --- |
| `all-pass` | First type fails band1 energy range, second passes all five criteria | `42,true` |
| `band1-range-fail` | Band 1 energy outside window rejects | `0,false` |
| `band-diff-fail` | Band energy difference below 15 dB requirement rejects | `0,false` |
| `peak-pos-only-pass` | Peak frequency position alone matches | `11,false` |
| `peak-pos-only-fail` | Peak frequency outside 11-12 kHz window rejects | `0,false` |
| `peak-width-only-fail` | Peak width criterion alone rejects (width computed from search-range peak even without the position check) | `0,false` |
| `mean-freq-only-pass` | Mean frequency alone matches | `12,false` |
| `length-only-fail` | Click length outside 0.05-0.10 ms window rejects | `0,false` |
| `length-zero-max-pass` | `clickLength[1] == 0` skips the length criterion entirely, so the type matches | `13,false` |
| `order-first-wins` | Two passing types; first in list wins | `21,false` |
| `no-selections-pass` | No criteria enabled means the type always matches | `30,false` |

The `length-zero-max-pass` and `no-selections-pass` rows pin PAMGuard semantics that would be easy to "fix" incorrectly in a port.

## Checker changes

The fixture reader now parses the multi-row `case,clickType,discard` schema and enforces exact case count and order against the C++ catalogue, so the fixture and catalogue cannot drift apart silently. The existing bad-config, empty-classifier, no-match, and standard-preset checks run unchanged.

## Validation

- Fixture regenerated via `generate-click-basic-classifier-fixture.ps1` (unchanged script); all eleven Java outcomes matched the intended per-case discrimination.
- `click_basic_classifier_parity` passes with all eleven decisions reproduced exactly by the C++ classifier.
- Full CTest suite passes `47/47` on Windows Release.

## Claim boundary

PAMGuard's `ClickTypeParams` defines exactly the five criteria flags covered here, so the criteria set is complete. The exporter mirrors the PAMGuard decision loop using PAMGuard's own transform classes (`RawDataTransforms.getComplexSpectrumHann`, `PamUtils.getMinFftLength`) rather than driving the GUI-coupled `BasicClickIdentifier` class directly; project/config import of real PAMGuard classifier settings remains a separate gap.
