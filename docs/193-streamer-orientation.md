# Streamer Orientation

Date: 2026-07-21

## Purpose

Completes the streamer geometry work from `docs/190`, which folded streamer *positions* into hydrophone coordinates but carried heading as unapplied metadata. Sessions can now describe a streamer's attitude — heading, pitch, and roll — and have hydrophone coordinates oriented by it, which is what PAMGuard does for any array whose streamer is not aligned with the world axes.

## Reference semantics ported

`docs/190` matched `PamArray.getAbsHydrophoneVector`, which is a translation only. That is genuinely all that method does, but it is not the whole story: `HydrophoneLocator.getPhoneLatLong` builds a `PamQuaternion` from the streamer's heading, pitch, and roll, rotates the hydrophone's coordinate vector by it with `PamVector.rotateVector`, and only then offsets by the streamer position:

```java
PamQuaternion streamerQuaternion = new PamQuaternion(Math.toRadians(streamerHead),
    Math.toRadians(streamerPitch), Math.toRadians(streamerRoll));
PamVector hydrophoneVec = new PamVector(hyd.getCoordinates());
hydrophoneVec = PamVector.rotateVector(hydrophoneVec, streamerQuaternion);
hydPos = hydPos.addDistanceMeters(hydrophoneVec);
```

Two details of that chain are convention-sensitive and worth stating, because getting either backwards produces a plausible-looking but wrong array:

- **The heading is flipped.** `PamQuaternion.fromEulerAngles` starts with `heading = 2*pi - heading`, so a positive heading turns **clockwise** — a compass bearing, not a mathematical angle. In the fixture, a hydrophone 10 m along `+y` at heading 90° lands on `+x`, not `-x`.
- **The Euler order is pitch, roll, heading**, mapped onto the quaternion's half-angle products as `c1/s1 = pitch/2`, `c2/s2 = roll/2`, `c3/s3 = heading/2`. This is not the aerospace yaw-pitch-roll ordering, so the combined-angle cases differ from a textbook composition; only the fixture settles it.

## Implementation

`pamguard::localisation::rotate_by_streamer_orientation` ports `fromEulerAngles` and `toRotation` verbatim — the same half-angle products, the same nine matrix terms — and applies the result to a vector. It takes radians; the callers convert.

`AnalysisSession`'s `resolve_streamer_offsets` now rotates each hydrophone's coordinates by its streamer's quaternion **before** adding the streamer position, matching `getPhoneLatLong`'s order. As in `docs/190`, this happens once at session construction, so every downstream consumer still sees plain absolute positions and no localisation path gained a branch.

`array.streamers[].pitchDegrees` and `rollDegrees` join the existing `headingDegrees` in session config; all three default to `0`, and all three are rejected if non-finite. Zero angles give the identity rotation, so **every existing session config is unaffected** — this is a config addition with no result-schema change, and the schema stays at v15.

## Validation

`streamer_orientation_parity` (new) replays `tests/fixtures/localisation/streamer-orientation.csv`, generated by `StreamerOrientationFixtureExporter` driving the **real** `pamMaths.PamQuaternion` and `PamVector.rotateVector`. Eight cases: identity, heading at 90°/180°/45°, isolated pitch and roll, and two combined attitudes including a negative heading with negative roll. Every component matches within `1e-12`.

The isolated-axis cases are the ones that pin the conventions: `heading-90` catches a heading-sign flip, `pitch-30` and `roll-30` catch a swapped Euler pair, and `combined-negative-heading` catches an ordering error that the single-axis cases would let through.

`session_lsq_bearing_wiring` gains a case asserting that a streamer with a non-trivial attitude *and* a non-zero position localises identically to the same array with the rotation and offset pre-applied by hand. Because the offset is non-zero, that also pins the rotate-then-translate **order** — translating first would rotate the array about the wrong centre and move the bearings. The existing translation-equivalence case had its heading set to zero, since it asserts a streamer-relative layout matches an un-rotated absolute one and that is now only true without rotation.

Full CTest suite passes `67/67`.

## Claim boundary

Static attitude only. A streamer has one heading, pitch, and roll for the session's lifetime; PAMGuard's time-varying streamer data units — a towed array whose attitude changes through a tow, driven by GPS and heading sensors — remain unported, as `docs/190` already noted for positions. The engine also has no sensor-feed path, so nothing updates attitude from live data.

The rotation is the only part of `getPhoneLatLong` ported. That method's surrounding work is geographic — it resolves a streamer's `LatLong` from GPS and returns an absolute earth position — and the engine works in a local metre frame throughout, so the geographic half has no meaning here and is deliberately excluded.
