# PAMGuard Enterprise Port

A C++ engine and web service that reproduces selected [PAMGuard](https://www.pamguard.org/) passive-acoustic-monitoring maths — FFT/spectrogram, click detection, click train tracking (MHT), whistle/moan detection, and multi-channel bearing localisation — behind an HTTP API, with parity against the PAMGuard Java source proven by golden fixtures generated from PAMGuard's own classes.

The Java PAMGuard source in the parent directory (version **2.02.18b**) is the reference oracle. This is not a loose rewrite: every ported algorithm is either pinned to fixtures generated by running the *real* PAMGuard classes, or its divergence is documented explicitly. Correctness comes before convenience, and honesty about gaps comes before feature claims.

**Current state: 74/74 CTest checks green · result schema v22 · 50+ numbered work-package docs (`docs/154`–`docs/207`) each recording one validated slice with an explicit claim boundary · measured at 26.7× realtime across 50 detector-loaded sessions.**

---

## What exists

| Component | Description |
| --- | --- |
| **C++ engine** (`cpp-engine/`) | Static library of ported PAMGuard maths: windowing, real FFT, spectrogram; click trigger/features/basic classifier; click train tracking (simple ICI tracker *and* the full ported MHT stack: kernel, all seven chi2 variables, garbage reclamation, electrical-noise filter, classifier chain); whistle peak detector and connected-region tracker; correlation delays, pair/LSQ/ML-grid bearing localisers, array shape/direction semantics, streamer geometry and orientation, world/earth-frame direction vectors. |
| **HTTP service** (`pamguard_engine_service`) | Multi-session REST service: create sessions from JSON config, stream raw PCM in, get versioned JSON results out. API-key auth, owner/tenant metadata, result archiving with indexed detector-event queries, CSV export, Prometheus metrics, audit logging, session persistence. |
| **Web UI** (`web-ui/`) | Single-file browser dashboard: session create/list, spectrogram preview, whistle contour overlay, detector config incl. click classifier JSON, archive querying and export. Served by the engine service. |
| **Ingest** | `ffmpeg_stream_ingest` bridge (WAV/MP3/Icecast/BUTT-style sources → PCM POSTs, restart/backoff, realtime pacing) plus a Python multi-source supervisor (`ops/ingest_supervisor.py`). |
| **Project import** | `PamguardProjectConverter` (JVM-side, in `reference-tools/`): converts a real PAMGuard `.psfx` settings file into an engine session JSON — acquisition, array geometry (incl. streamers, per-axis errors), FFT, click detector, basic classifier types, MHT click-train settings, classifier chain, whistle settings. |
| **Reference tooling** (`reference-tools/`) | ~30 Java fixture exporters that drive real PAMGuard classes and emit the CSV fixtures the C++ parity tests replay; PowerShell generators for each. |
| **Ops & deploy** | Archive retention and event-index rebuild utilities (`ops/`), Dockerfile + compose file, starter Kubernetes manifests (`deploy/kubernetes/`), Windows GitHub Actions workflow. |

## What works, and how well

Three tiers, tracked per-module in the **[parity ledger](docs/126-pamguard-parity-ledger.md)** (the authoritative table — read it before relying on any specific module):

- **Fixture parity** — validated against CSVs generated by running the real PAMGuard Java classes, typically to 1e-12 or better (several are exact or ~1e-17): windows, FFT, spectrogram chunking, click trigger/classifier, IDI statistics, all MHT chi2 variables, MHT kernel and full StandardMHTChi2 stack, spectrum templates, whistle contour delays, pair/LSQ/ML-grid bearing localisers, array shape semantics, streamer orientation, world and earth-frame vectors.
- **Foundation** — implemented with focused regression tests but not yet full PAMGuard equivalence: click train tracker end-to-end, whistle localisation service path, far-field bearings, train summaries.
- **Operational** — service/deployment features that are not detector maths: archiving, multi-session smoke (50 sessions), ingest supervision.

Deliberately preserved PAMGuard quirks (tests pin them so they can't silently "get fixed"): the frequency-overlap `max`-instead-of-`min` in whistle grouping, the time-delay chi2 drop-worst-pair rule, the theta grid that never reaches +π, `Math.toRadians`' one-ulp constant, and others — each documented in its slice doc.

## What is *not* done

Honesty section. These are recorded decisions or genuinely blocked items, not oversights:

1. **Deliberate non-ports (with evidence):** `SimplexBearingLocaliser`, `CombinedBearingLocaliser`, `MLGridBearingLocaliser` v1 (zero live call sites in the whole PAMGuard tree — [docs/200](docs/200-remaining-bearing-localisers-non-port.md)); `DetectionGroupLocaliser` (`@Deprecated`, needs a GPS/track model — [docs/192](docs/192-group-localiser-non-port.md)).
2. **Documented substitution:** grid-selected (plane/volume) sub-arrays report `bearingLocaliser: "grid"` and the ported ML grid localiser runs, but `MLLineBearingLocaliser2` is only available as an unselected config flag because PAMGuard gates it behind an SMRU licence ([docs/199](docs/199-ml-line-bearing-variant.md)).
3. **Blocked on external data** (the complete list, [docs/204](docs/204-per-chunk-attitude.md)):
   - `.psfx` import from a **different** PAMGuard build — the converter is pinned to 2.02.18b; Java serialisation is version-brittle, and testing another version needs a file written by it.
   - Time-varying array **geometry** (towed-array deformation), **geographic position** (`LatLong`), and sub-chunk attitude interpolation — all need a live GPS/attitude feed. Static geometry, static + per-chunk attitude, and earth-frame vectors are done.
4. **Scale:** throughput is now measured — 50 detector-loaded live sessions sustain **26.7× realtime** (p95 chunk latency 48 ms) on one workstation via `service-throughput-bench.ps1` ([docs/207](docs/207-throughput-benchmark.md)). Multi-hour soak, archive-enabled throughput, and concurrent-client runs remain unmeasured.
5. Bearings are directions, not positions: there is no target-motion analysis or 3D localisation.

## Installation

### Prerequisites (Windows; Linux via the Dockerfile)

| Tool | Used for | Notes |
| --- | --- | --- |
| Visual Studio 2022 Build Tools (MSVC, C++20) | building the engine | `vcvars64.bat` must be on a known path |
| CMake ≥ 3.20 + Ninja or VS generator | build system | |
| JDK 21 (Temurin tested) | reference tools: fixture generation, project import | only needed to *regenerate* fixtures or import `.psfx`; checked-in fixtures build/test without it |
| Maven (or `reference-tools/scripts/mvn-local.ps1`) | compiling the parent PAMGuard Java source to `target/classes` | same caveat — only for reference tooling |
| Python 3 | ops utilities, one generated header check | |
| FFmpeg | live-stream ingest bridge | only for ingest |

The repo layout assumes this workspace sits **inside a PAMGuard source checkout** (the parent directory is the Java reference; reference tools compile against `../target/classes`).

### Build and test

```powershell
# From a VS 2022 x64 developer prompt (or after calling vcvars64.bat):
cd pamguard-enterprise-port\cpp-engine
cmake -S . -B build
cmake --build build --config Release
ctest --test-dir build -C Release --output-on-failure   # expect 74/74
```

`scripts/build-msvc.ps1` and `scripts/test-msvc.ps1` wrap the same steps. The HTTP smoke (`scripts/service-smoke.ps1`) and 50-session load smoke (`scripts/service-load-smoke.ps1`) run as part of CTest but can be invoked directly.

## Running the service

```powershell
# Minimal: port 8080, no auth, in-memory sessions
.\build\Release\pamguard_engine_service.exe 8080

# Production-ish: auth, persistence, archiving, web UI, OpenAPI
$env:PAMGUARD_API_KEY            = "change-me"        # or PAMGUARD_API_KEY_FILE
$env:PAMGUARD_SESSION_CONFIG_DIR = "C:\data\sessions" # session persistence across restarts
$env:PAMGUARD_RESULT_ARCHIVE_DIR = "C:\data\archive"  # result + detector-event archiving
$env:PAMGUARD_WEB_UI_FILE        = "..\..\web-ui\index.html"
$env:PAMGUARD_OPENAPI_FILE       = "..\..\platform\openapi.yaml"
$env:PAMGUARD_REQUIRE_SESSION_METADATA = "1"          # enforce ownerId/tenantId
.\build\Release\pamguard_engine_service.exe 8080
```

Other knobs: `PAMGUARD_MAX_SESSIONS`, `PAMGUARD_MAX_PCM_BODY_BYTES`, `PAMGUARD_MAX_ARCHIVE_QUERY_RECORDS`, `PAMGUARD_HTTP_THREADS`, `PAMGUARD_CORS_ORIGIN`, `PAMGUARD_AUDIT_LOG_FILE`, `PAMGUARD_INGEST_STATUS_FILE`.

### Minimal end-to-end example

```powershell
# 1. Create a two-channel session with click detection + localisation
$session = @'
{
  "sessionId": "demo", "sourceId": "demo-src",
  "sampleRateHz": 48000, "channelCount": 2,
  "fft": { "length": 512, "hop": 256, "windowType": "Hann", "channels": [0, 1] },
  "array": {
    "id": "demo-array", "speedOfSoundMps": 1500.0,
    "hydrophones": [
      { "channel": 0, "xM": 0.0, "yM": 0.0, "zM": 0.0 },
      { "channel": 1, "xM": 1.0, "yM": 0.0, "zM": 0.0 }
    ]
  },
  "click": { "enabled": true, "localisation": true, "thresholdDb": 10.0 }
}
'@
Invoke-RestMethod -Method Post -Uri http://localhost:8080/sessions -ContentType application/json -Body $session

# 2. Stream interleaved little-endian float32 PCM at it (repeat per chunk)
Invoke-RestMethod -Method Post `
  -Uri "http://localhost:8080/sessions/demo/pcm-f32le?startSample=0" `
  -ContentType application/octet-stream -InFile chunk.f32le
# The response is the full versioned result JSON for that chunk:
# clicks, clickLocalisations (delays, pair/LSQ/grid bearings, world vectors),
# clickTrains, whistle regions/delays/groups, schemaVersion, continuity info.

# 3. Optional per-chunk array attitude (holds until replaced):
#    ...pcm-f32le?startSample=N&headingDegrees=42&pitchDegrees=-7&rollDegrees=11

# 4. Flush, inspect, query the archive, delete
Invoke-RestMethod -Method Post   -Uri http://localhost:8080/sessions/demo/flush
Invoke-RestMethod -Method Get    -Uri "http://localhost:8080/sessions/demo/archive/detections?type=click&limit=10"
Invoke-RestMethod -Method Delete -Uri http://localhost:8080/sessions/demo
```

Full API surface: **[platform/openapi.yaml](platform/openapi.yaml)** (also served at `/openapi.yaml` when configured). Result fields are versioned — see **[docs/74-result-schema-version.md](docs/74-result-schema-version.md)** for the v1→v21 history. The web UI is served at `/` when `PAMGUARD_WEB_UI_FILE` is set.

## Importing a PAMGuard project (.psfx)

Runs on the JVM because `.psfx` payloads are Java-serialised object graphs ([why](docs/182-project-import-feasibility.md)):

```powershell
# Compile PAMGuard + reference tools first (once):
.\reference-tools\scripts\mvn-local.ps1 -DskipTests compile

# Convert a settings file to an engine session JSON:
java -cp "<build>;<target/classes>;<deps>" org.pamguard.port.reference.PamguardProjectConverter `
     convert MyProject.psfx session.json
# ...then POST session.json to /sessions (add ownerId/tenantId if enforced).
```

`reference-tools/scripts/generate-project-import-fixture.ps1` shows the exact classpath incantation, and the HTTP smoke round-trips the checked-in sample end to end. Everything the engine supports is mapped; everything else is **reported and skipped, never silently dropped**. Files from PAMGuard builds other than 2.02.18b may fail to deserialise (exit 3) — see [docs/203](docs/203-project-import-converter.md).

## Live ingest

```powershell
# Single source: FFmpeg bridge posts PCM chunks with realtime pacing + restart/backoff
.\build\Release\ffmpeg_stream_ingest.exe --help

# Many sources: supervised worker per source from a JSON manifest
python ops\ingest_supervisor.py --manifest platform\ingest-sources.example.json --status-file status.json
```

## Regenerating parity fixtures

Fixtures are checked in; regenerate only when deliberately re-pinning against the Java oracle:

```powershell
.\reference-tools\scripts\mvn-local.ps1 -DskipTests compile
.\reference-tools\scripts\generate-<name>-fixture.ps1   # one per fixture family
```

Each generator compiles a small exporter that drives the *real* PAMGuard classes. After regenerating, `git status` must show no unexplained diffs to pinned fixtures.

## Development rules

- PAMGuard correctness/parity first; every change is an enterprise work package with its own doc.
- Behaviour or API output changes update the OpenAPI spec and bump the result schema version (asserted in three places by the smoke).
- New maths gets a fixture generated from the real Java classes wherever `PamController` coupling allows; where it doesn't, branch-coverage tests and an honest note in the ledger.
- Preserve reference quirks; pin them with tests so they cannot silently change.
- No destructive Git operations; commit small validated checkpoints.
